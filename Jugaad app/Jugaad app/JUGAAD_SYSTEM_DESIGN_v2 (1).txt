🧠  PHASE 0 - HARD SYSTEM RULES (NEVER VIOLATE)

These 7 rules apply to EVERY service, EVERY function, EVERY write. If any rule is violated, debug hell follows.

• Firestore is the ONLY source of truth. No other DB holds canonical state.
• Every state change MUST increment job.version. No version bump = silent bug.
• No service writes events directly - ONLY via outbox subcollection.
• Redis is NON-CRITICAL - treat as cache only. App must work without it (degraded).
• All async handlers MUST be idempotent - handle duplicate events gracefully.
• All APIs MUST be backward compatible - never break old client versions.
• Deployment MUST be canary-based - never push to 100% of users instantly.

⚡ MVP SIMPLIFICATION: Redis geo is dropped. Matching uses Firestore geo queries only. Add Redis after 500+ concurrent users.

⚡  PHASE 3 - GEO MATCHING (FIRESTORE-ONLY, NO REDIS)

Redis geo is REMOVED from MVP. Reason: Redis adds operational complexity (eviction, crash recovery) that kills a solo founder. Firestore geo queries handle <500 concurrent users fine.

3.1 Geo Hash Strategy
• Use geohash-js library to encode lat/lng as geohash string
• Store geohash on worker document, updated every heartbeat (30s)
• Query workers by geohash prefix to find nearby workers
• Precision level 6 geohash ≈ 1.2km radius - good for urban matching

3.2 Matching Algorithm (Steps in Order)

Step
Action
Debug checkpoint
1. Geo query
Find workers with matching geohash prefix (precision 5 = ~5km)
Log: workers_found_geo_count
2. Skill filter
Keep only workers with matching skill
Log: workers_after_skill_filter
3. Status filter
Keep only online + not busy workers. Check last_heartbeat < 90s ago.
Log: workers_after_status_filter
4. Approval filter
Keep only approved workers
Log: workers_after_approval_filter
5. Score & rank
Score = (1/distance_km * 0.5) + (rating/5 * 0.3) + (1 - missed_rate * 0.2)
Log: top_workers with scores
6. Select top 5
Take top 5 scored workers
Log: selected_workers array
7. Broadcast
Write JOB_BROADCASTED outbox event. All 5 workers get FCM push.
Log: broadcast_sent_count
8. First accept wins
Transaction locks job to first worker who calls /accept
Log: accepted_by_worker_id

FALLBACK: If workers_after_status_filter == 0 at step 3 → set fallback_triggered = true on job → UI shows Matching A2 screen (schedule/callback options) instead of error screen.

3.3 Heartbeat System
• Worker app sends POST /v1/workers/{id}/heartbeat every 30 seconds
• Heartbeat updates: lat, lng, geo_hash, last_heartbeat timestamp
• Matching service checks: last_heartbeat > now - 90s → worker is online
• No Redis. No TTL eviction. Simple timestamp comparison in Firestore.

4.3 Outbox Publisher Pattern
Publisher runs as always-on Cloud Run service. Uses Firestore CDC (Change Data Capture) via Eventarc.

• Firestore write triggers Eventarc → Cloud Run Publisher
• Publisher reads outbox document → publishes to Pub/Sub → marks outbox status = published
• If Pub/Sub fails → status = failed, retry_count++ → alert if retry_count > 3
• Consumer checks event.version >= job.version before processing. Ignores stale events.

DEBUG TIP: If an event is not processed, check /jobs/{id}/outbox → look for status=failed or retry_count > 0. This is your audit trail.

Up to 5 workers receive the broadcast. ALL 5 may tap Accept simultaneously. Firestore transaction ensures only ONE wins.

Step
Action

401

5.3 Worker Acknowledge - POST /v1/jobs/{jobId}/ack
• Called when worker taps 'I've arrived' in app
• Transaction: if job.expires_at < now → return 410 Gone (timeout). Else: worker_ack=true, status=in_progress, version++
• Write outbox JOB_ACK event → cancels the Cloud Task timeout

5.4 Timeout Handler - Cloud Tasks
• Fires at expires_at time (120 seconds after assignment)
• Handler reads job: if worker_ack == false AND status == assigned → revert to searching
• Revert: status=searching, worker_id=null, version++, write outbox JOB_TIMEOUT
• Matching service picks up JOB_TIMEOUT and runs matching again with different worker pool
• If 3 consecutive timeouts → set fallback_triggered=true → UI shows Matching A2

5.5 Fallback Trigger - Admin Manual Assignment
• Admin ops panel calls POST /v1/admin/jobs/{jobId}/assign with worker_id
• Transaction: update job with worker_id, status=assigned, is_manual_assign=true, version++
• Write outbox JOB_MANUAL_ASSIGN → Notification Service sends SMS to both user and worker
• Worker receives FCM notification with job details

10.2 Deployment Order (ALWAYS follow this order)
• 1. Firestore schema changes first (backward compatible - new fields, never delete fields)
• 2. Consumer services second (they now handle new events but old events also work)
• 3. Producer services third (they now emit new events that consumers handle)
• 4. Mobile app canary last (triggers new flows on backend that is already ready)

10.3 Rollback Procedure
• Cloud Run: traffic split → revert instantly in Cloud Console (< 30 seconds)
• Firestore: data changes are irreversible. Design schemas to be forward-compatible only.
• Pub/Sub: old events replay automatically. Ensure consumers handle all past versions.

🧠  PHASE 12 - DEBUG RUNBOOK (MOST COMMON ISSUES)

12.1 'No workers found' happening too often
• Check workers_after_status_filter log → if always 0, workers are offline
• Check worker.last_heartbeat in Firestore → if stale, Flutter heartbeat is broken
• Check worker.approval_status → if pending, admin hasn't approved them yet
• Check geo_hash precision → if mismatch between job and worker geo hash, no matches

12.2 Job stuck in 'assigned' status
• Check worker_ack value → if false and expires_at < now, Cloud Task should have fired
• Check Cloud Tasks queue → if task is missing, it failed to enqueue (check Cloud Run logs)
• Fix: manually update job.status=searching and version++ via Firestore console

12.3 Pub/Sub events not being processed
• Check /jobs/{id}/outbox → status field should be 'published'
• If status = 'pending' for > 5 min → Eventarc → Cloud Run Publisher is broken
• Check Cloud Run Publisher logs in Cloud Logging for errors
• Check if Eventarc trigger is active in Cloud Console

12.4 Payment webhook not arriving
• Check Razorpay dashboard → Webhooks section → delivery status
• Check Cloud Run /v1/webhooks/razorpay endpoint is live and returning 200
• Test with Razorpay sandbox → manually trigger test webhook
• Check signature verification code → ensure webhook secret matches Razorpay dashboard

12.5 Flutter UI showing wrong job state
• Check Firestore job doc directly → status + version are ground truth
• Firestore listener may have been disconnected → call .listen() reconnect on app foreground
• If job.version on client < Firestore version → force full sync (fetch job doc fresh)
